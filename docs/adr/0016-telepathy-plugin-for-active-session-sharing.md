# 0016. Telepathy plugin for active session context sharing

**Status**: Accepted (Web UI integration superseded by #247; plugin and MCP tools remain in effect)

**Date**: 2026-05-23

## Context

Kizuna captures memories at session end (via the SessionEnd hook) and recalls them at session start (via UserPromptSubmit). This means there is no mechanism for sharing context between **active** Claude Code sessions running in parallel.

In multi-repository development workflows — such as a frontend and backend team working on the same product — it is common to investigate an issue on one side and want to immediately share findings with the other. Currently, this requires manually copying and pasting text between terminals, which is tedious and loses context formatting.

### Use case

A production incident occurs. The user investigates and brainstorms fixes in the frontend session. They want to share this context with the backend session so both sides can coordinate. The user says "telepathize this" (「テレパシーして」) to the frontend Claude, which summarizes the conversation and sends it. The backend Claude retrieves the message when the user says "receive telepathy" (「テレパシーを受け取って」).

The mental model is **copy-paste between terminals**, automated through Claude.

### Design constraints

- Must comply with the local-first, single-SQLite-file principle (ADR-0001)
- Must not write to another project's database (ADR-0013 established read-only access to referenced databases)
- Must use the same `references` configuration format as `multi-repo-sharing` for database path resolution (but configured independently per plugin for isolation)
- Must not require LLM in the core (ADR-0007) — summarization is performed by Claude in the session, not by Kizuna

## Decision

Implement a **telepathy plugin** (`packages/plugin-telepathy/`) that provides two MCP tools for explicit, user-initiated context sharing between active sessions.

### Architecture

#### Storage

A dedicated `telepathy_messages` table in each project's own `memory.db`, created via plugin migration:

```sql
CREATE TABLE telepathy_messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  message TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Each project stores only its own outgoing messages. Only the latest message is retained (see Lifecycle below).

#### MCP tools

- **`kizuna_telepathy_send(message: string)`**: Deletes any existing telepathy message in the local database, then inserts the new message. The `message` parameter contains a summary generated by Claude in the active session.

- **`kizuna_telepathy_receive()`**: No parameters. Opens each referenced project's database (from the `multi-repo-sharing` `references` configuration) in read-only mode, reads the latest telepathy message from each, and returns all found messages with source attribution.

#### Routing

There is no recipient addressing. Messages are broadcast: the sender writes to its own database, and any project that has a `reference` pointing to the sender's database can read the message. Whether the message is relevant is determined by the user, not the system.

#### Lifecycle

Each project retains at most **one** telepathy message. When `kizuna_telepathy_send` is called, the previous message is deleted before the new one is inserted. This eliminates the need for TTL, read markers, maintenance integration, or garbage collection.

#### Database access pattern

- **Write**: Only to the local project's `memory.db` (on send)
- **Read**: Local database + referenced databases in read-only mode (on receive)

This is consistent with the federated search pattern established in ADR-0013.

## Rationale

### Why a plugin, not core

Telepathy is an optional capability that depends on `multi-repo-sharing`'s `references` configuration. Not all users need cross-project communication. The plugin architecture (ADR-0005) exists precisely for this kind of opt-in extension.

### Why explicit send/receive, not automatic sync

Automatic synchronization of active session state would generate excessive noise and remove user control over what gets shared. The explicit model lets users control:

- **When** to share (timing)
- **What** to share (Claude summarizes on instruction)
- **Whether** to receive (pull, not push)

### Why one message per project

The use case is ephemeral context transfer — a clipboard, not a mailbox. Retaining history adds complexity (pagination, cleanup) without clear benefit. If the context is important enough to persist, it will naturally be captured by Kizuna's normal session-end memory system.

### Why manual refresh only (no polling)

> **Note (#247)**: The Web UI telepathy page described in this section was removed in #247. Telepathy is now used exclusively via the plugin and MCP tools (`kizuna_telepathy_send` / `kizuna_telepathy_receive`); FE↔BE handoff happens through MCP, so a Web UI view was no longer needed. The rationale below is retained as the historical record of the original Web UI design decision.

The Web UI's telepathy receive section uses manual refresh exclusively. Automatic polling was considered and rejected because:

- Telepathy messages are ephemeral and infrequent — they change only when a user explicitly sends from another project
- Polling would add unnecessary network/database load for a feature that is inherently pull-based
- The user controls when to check for messages, consistent with the explicit send/receive design

### Why no recipient addressing

Addressing adds configuration overhead and limits the broadcast use case. In a typical multi-repo setup with 2–3 projects, receiving all messages (at most one per project) is manageable. The user can quickly scan and decide relevance.

## Consequences

### Positive

- Enables real-time context sharing between active Claude Code sessions
- Familiar configuration: uses the same `references` format as `multi-repo-sharing`
- Minimal storage impact (one row per project in `telepathy_messages`)
- Simple mental model: "copy-paste between terminals via Claude"
- Consistent with existing federated read-only access pattern (ADR-0013)

### Negative

- Requires its own `references` to be configured in the plugin options — these are independent from `multi-repo-sharing`'s references, so users must configure them separately
- One message per project means rapid successive sends overwrite previous messages without warning
- No delivery guarantee — if the receiving session never calls `receive`, the message sits until overwritten
- Referenced database paths must be valid on the local filesystem (same constraint as ADR-0013)

### Constraints introduced

- The `telepathy_messages` table is plugin-managed; core Kizuna has no awareness of it
- The plugin must open referenced databases in read-only mode; write access to referenced databases is prohibited
- Summarization quality depends entirely on Claude's in-session capability; Kizuna does not validate or transform the message content
