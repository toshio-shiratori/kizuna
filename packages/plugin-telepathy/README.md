# @kizuna/plugin-telepathy

Kizuna plugin for real-time context sharing between active Claude Code sessions. Provides MCP tools to send and receive short messages across projects, enabling coordinated multi-repository workflows.

See the [root README](../../README.md) for full project context.

## Installation

```bash
pnpm add @kizuna/plugin-telepathy
```

Requires `@kizuna/core` and `better-sqlite3` as peer dependencies. Requires Node.js >= 24.0.0.

## Configuration

Add to your `.kizuna/plugins.json`:

```json
{
  "plugins": {
    "telepathy": {
      "enabled": true,
      "options": {
        "references": [
          {
            "name": "backend-api",
            "dbPath": "/path/to/backend-api/.kizuna/memory.db"
          }
        ]
      }
    }
  }
}
```

Each entry in `references` declares another project's database to read telepathy messages from. The `name` field is used for source attribution in received messages.

References are **directional** -- if project A references project B, A can receive messages from B, but B cannot receive from A unless it also configures a reference to A.

## How It Works

The plugin adds two MCP tools to the Kizuna MCP server:

### `kizuna_telepathy_send(message)`

Writes a telepathy message to the local project's database. Any previous message is deleted first -- each project retains at most **one** message at a time.

The `message` parameter is a free-form string, typically a conversation summary generated by Claude in the active session.

### `kizuna_telepathy_receive()`

Opens each referenced project's database in read-only mode and reads the latest telepathy message from each. Returns all found messages with source attribution and timestamps.

## Usage Example

In the frontend session:

> "This auth bug is caused by an expired token refresh race condition. Telepathize this to the backend team."

Claude calls `kizuna_telepathy_send` with a summary of the investigation.

In the backend session:

> "Receive telepathy from the frontend."

Claude calls `kizuna_telepathy_receive` and gets the frontend's context.

## Design

- **One message per project** -- sending overwrites the previous message. This is a clipboard, not a mailbox.
- **No recipient addressing** -- messages are broadcast. Any project with a reference can read them.
- **Read-only remote access** -- referenced databases are opened with `{ readonly: true }`. The plugin never writes to another project's database.
- **No delivery guarantee** -- if the receiving session never calls `receive`, the message sits until overwritten.

See [ADR-0016](../../docs/adr/0016-telepathy-plugin-for-active-session-sharing.md) for the full design rationale.

## Options

| Option       | Type              | Default | Description                                                 |
| ------------ | ----------------- | ------- | ----------------------------------------------------------- |
| `references` | `RepoReference[]` | `[]`    | List of other projects' databases to receive messages from. |

### RepoReference

| Field    | Type     | Description                                         |
| -------- | -------- | --------------------------------------------------- |
| `name`   | `string` | Display name for source attribution in messages     |
| `dbPath` | `string` | Absolute path to the referenced project's memory.db |

## Error Handling

- Referenced databases that do not exist are skipped with a warning log.
- Referenced databases without a `telepathy_messages` table are skipped with a debug log.
- Errors from individual references do not affect the retrieval of other references.

## Exports

- `createTelepathy()` -- Factory function returning a fresh plugin instance.
- `telepathy` -- Pre-configured plugin instance for convenience.
- `sendMessage(db, message)` -- Low-level send function.
- `receiveMessages(references, logger)` -- Low-level receive function.
- `hasTelepathyTable(db)` -- Schema compatibility check utility.
